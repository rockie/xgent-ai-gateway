## Unsatisfied Requirements

### LIFE-04: Failed tasks retry with configurable max retries and exponential backoff
- **Phase:** 4 (Task Reliability and Callbacks)
- **Status:** Not implemented
- **Evidence:** `max_retries` field exists in `ServiceConfig` but the reaper marks tasks as failed immediately. No retry counter exists, no re-enqueue logic.
- **REQUIREMENTS.md:** `[ ]` Pending — correctly marked

### LIFE-05: Tasks exhausting retries move to a per-service dead letter queue
- **Phase:** 4 (Task Reliability and Callbacks)
- **Status:** Not implemented
- **Evidence:** No DLQ key pattern, no DLQ write logic, no DLQ read API exists anywhere in the codebase.
- **REQUIREMENTS.md:** `[ ]` Pending — correctly marked

### NODE-02: Internal nodes can reverse-poll via HTTPS (partial)
- **Phase:** 1 (Core Queue Loop)
- **Status:** Deferred by design decision D-13
- **Evidence:** Runner agent proxy model replaces HTTP node polling with gRPC-only. No HTTP poll endpoint exists. `REQUIREMENTS.md` marks `[x] Complete` which is misleading.

## Cross-Phase Integration Issues

### Critical

1. **gRPC TaskService has no API key authentication** — `api_key_auth_middleware` only applied to HTTP routes. Any caller on port 50051 can submit tasks and query status without credentials. Affects AUTH-01, TASK-01, RSLT-01.

2. **gRPC ReportResult/Heartbeat/DrainNode have no node token auth** — Only `PollTasks` validates node tokens. The other 3 RPCs accept unauthenticated requests. An unauthenticated caller can mark any task complete, spoof heartbeats, or trigger drain. Affects AUTH-03, NODE-03, NODE-04, NODE-06.

### Significant

3. **`in_flight_tasks` counter never decremented** — `update_in_flight_tasks(delta=1)` called on poll dispatch, but `report_result` never calls `update_in_flight_tasks(delta=-1)`. Counter grows monotonically. Admin health dashboard (`/v1/admin/health`) shows permanently inflated node workload. Affects NODE-05, OBSV-03.

### Minor

4. **gRPC proto lacks `callback_url` field** — gRPC clients cannot set per-task callback URLs (only HTTP or per-key default). Affects RSLT-03 for gRPC clients.

5. **Revoke routes use `POST .../revoke` instead of `DELETE .../{hash}`** — API contract deviation, functionally correct.

6. **Plain HTTP mode has no keepalive config** — HTTP/2 keepalive only on TLS path. INFR-06 partial in dev mode.

## Broken Flows

### Flow 1: gRPC Client Submit → Result Retrieval
- Breaks at: Auth check — no API key interceptor on gRPC TaskService
- A client can submit tasks to any registered service over gRPC without credentials

### Flow 2: Node Reports Result → Health Update
- Breaks at: `in_flight_tasks` counter not decremented after `report_result`
- Health dashboard shows permanently inflated workload counts for active nodes

## Complete Flows

1. **HTTP Client Full Lifecycle** — Submit (auth check → service check → enqueue) → Node polls (token check) → Node executes → Reports result → Client retrieves via HTTP GET → Callback delivered
2. **Node Drain** — SIGTERM → DrainNode RPC → No new tasks assigned → Complete in-flight → Exit
3. **Observability** — Gateway starts → Structured JSON logs → Metrics recorded → /metrics and /v1/admin/health available
