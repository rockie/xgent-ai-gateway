use std::collections::HashMap;

The enhanced dry-run block should:

**1. Keep existing config summary output** (service, mode, gateway, node_id, mode-specific fields).

**2. Add validation checks** (per D-10). Collect errors into a `Vec<String>` called `errors`:

For CLI mode:
- Check that `command[0]` (the binary/script path) exists on disk using `Path::new(&command[0]).exists()` AND is executable. To check executable on Unix: `use std::os::unix::fs::PermissionsExt;` then `std::fs::metadata(&command[0]).map(|m| m.permissions().mode() & 0o111 != 0)`. If the path does not exist, push error `"command not found: {path}"`. If exists but not executable, push error `"command not executable: {path}"`.
- Print: `"  Command check: {path} ... ok"` or `"  Command check: {path} ... NOT FOUND"` or `"  Command check: {path} ... NOT EXECUTABLE"`

For SyncApi mode:
- Parse `sync_api.url` with `Url::parse()`. If it fails, push error `"invalid sync-api URL: {url} ({parse_error})"`.
- Print: `"  URL check: {url} ... ok"` or `"  URL check: {url} ... INVALID ({err})"`

For AsyncApi mode:
- Parse `async_api.submit.url` with `Url::parse()`. If it fails, push error.
- Parse `async_api.poll.url` with `Url::parse()`. Note: poll URL may contain `<submit_response.X>` placeholders which make it not a valid URL. If the URL contains `<submit_response.` prefix, skip URL validation and print: `"  Poll URL check: {url} ... skipped (contains submit_response placeholders)"`. Otherwise validate normally.
- Print results for both submit and poll URLs.

**3. Add response template preview** (per D-09). After the validation checks, print the response body template with sample placeholder values:

Build a sample variables HashMap based on mode:
- CLI mode: `{"stdout": "(sample stdout)", "stderr": "(sample stderr)", "payload": "(sample payload)", "service_name": config.service.name}`
- SyncApi mode: `{"response.result": "(sample response.result)", "response.data": "(sample response.data)", "payload": "(sample payload)", "service_name": config.service.name}` -- Also scan the success body template with `find_prefixed_placeholders(&body, "response")` and for each path found, add `"response.{path}": "(sample response.{path})"` to the map.
- AsyncApi mode: scan with `find_prefixed_placeholders(&body, "poll_response")` and for each path, add `"poll_response.{path}": "(sample poll_response.{path})"`. Also add `"payload"` and `"service_name"`.

Call `resolve_placeholders(&config.response.success.body, &sample_vars)` and print:
```
  Response template preview:
    Success: {resolved_template}
```
If the config has a `failed` response section, do the same:
```
    Failed:  {resolved_failed_template}
```
If resolution fails (unresolved placeholder), print:
```
    Success: (unresolvable: {error})
```

**4. Print summary line** (per D-11):
- If `errors` is empty: print a blank line then `"  \u{2713} Config is valid"` (checkmark character)
- If `errors` is not empty: print each error as `"  ERROR: {msg}"`, then a blank line, then `"  \u{2717} Config has errors"` (X mark character), and `std::process::exit(1)`

The complete output flow:
```
Config loaded successfully from '{path}'
  Service:  {name}
  Mode:     {mode}
  Gateway:  {gateway}
  Node ID:  {node_id}
  [mode-specific fields]
  Command check: ./echo.sh ... ok          (CLI only)
  URL check: http://... ... ok              (sync/async only)
  Response template preview:
    Success: {"output": "(sample stdout)"}
    Failed:  {"error": "(sample stderr)"}

  [ERROR lines if any]

  ✓ Config is valid
```
