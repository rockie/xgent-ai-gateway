<objective>
Wire callback URL support into task submission, API key management, and result reporting. Trigger callback delivery from both report_result and the reaper on terminal state. Add integration tests.

Purpose: RSLT-03 and RSLT-04 require clients to optionally receive results via callback URL with retry. This plan adds callback_url to API keys and task submission, wires delivery triggers, and proves the system works end-to-end with Redis.
Output: Working callback delivery triggered on task completion/failure, PATCH endpoint for API key callback URL, integration tests.
</objective>

   The challenge: `report_result` currently only has access to `&amp;self` (RedisQueue), not AppState. To trigger callbacks, it needs the http_client and callback config. Two approaches:

   - (b) Add a method on RedisQueue that the caller uses after report_result.

   Use approach (a): Modify `report_result` return type from `Result&lt;(), GatewayError&gt;` to `Result&lt;Option&lt;String&gt;, GatewayError&gt;` where the Option&lt;String&gt; is the callback_url from the task hash (if present). Read `callback_url` from the HGETALL fields that report_result already reads. Return `Ok(fields.get("callback_url").cloned().filter(|s| !s.is_empty()))`.

   Then update all callers of `report_result`:
   - `gateway/src/grpc/poll.rs` — In the report_result gRPC handler, after calling `report_result`, if it returns `Some(url)`, spawn callback delivery:
     ```rust
     if let Ok(Some(callback_url)) = result {
         let client = state.http_client.clone();
         let tid = task_id.clone();
         let st = if success { "completed" } else { "failed" };
         let cfg = &state.config.callback;
         tokio::spawn(crate::callback::deliver_callback(
             client, callback_url, tid, st.to_string(),
             cfg.max_retries, cfg.initial_delay_ms,
         ));
     }
     ```

   let callback_url: Option<String> = redis::cmd("HGET")
