## Broken Flow: Node.js Client URL Mismatch

**Severity:** BLOCKING (EXMP-04 requirement unsatisfied)

All 3 Node.js client scripts (`cli-client.js`, `sync-api-client.js`, `async-api-client.js`) submit to `/api/v1/tasks` and poll `/api/v1/tasks/${task_id}`. The gateway registers routes at `/v1/tasks` — there is no `/api` prefix. Every fetch call returns HTTP 404.

**Fix:** Remove `/api` prefix from all 3 scripts and the README.

**Files to fix:**
- `examples/nodejs-client/cli-client.js` (lines 17, 42)
- `examples/nodejs-client/sync-api-client.js` (lines 17, 42)
- `examples/nodejs-client/async-api-client.js` (lines 20, 45)
- `examples/nodejs-client/README.md` (lines 62, 63, 71)

## Tech Debt: Header Forwarding is Inert

**Severity:** Non-blocking (body templates work correctly for all modes)

`ExecutionResult.headers` is populated by all 3 executors from `response.success.header` / `response.failed.header` config, but `ReportResultRequest` in `gateway.proto` has no `headers` field. The header data is silently dropped in `agent.rs`. Any operator who configures response headers will see no effect.

**Options:**
1. Add `map<string,string> headers` to `ReportResultRequest` proto and thread through gateway
2. Remove header config fields to avoid misleading operators
3. Document as known limitation and defer
